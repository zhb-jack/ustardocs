## 6. 本次修复的核心问题

### 6.1 Kong 到 TGO 的错误上游

之前 Kong 仍然把 TGO 指向：

- `host.docker.internal`

在测试服 Linux 环境下，这种方式不稳定，容易导致：

- TGO 页面卡住
- `/v1/setup/status` 异常
- `/v1/wukongim/route` 异常

当前已改为容器直连：

- `tgo-nginx`
- `tgo-api`
- `tgo-wukongim`

### 6.2 TGO API 对 WuKongIM 的旧主机名引用

根因：

- `tgo-api` 里 `WUKONGIM_SERVICE_URL` 还在写旧值 `http://wukongim:5001`
- 实际测试服当前可稳定解析的容器名为 `tgo-wukongim`

已修正为：

- `http://tgo-wukongim:5001`

### 6.3 TGO 客服回消息失败

原始报错：

- `Platform Service request error: [Errno -3] Temporary failure in name resolution`

根因分为两层：

1. `tgo-platform` 没有接入 `snow-tgo-net`，无法解析 `ustarpay-kong / ustarpay-auth-service`。
2. `custom` 平台配置里缺少：
   - `callback_url`
   - `platform_api_key`

最终修复方式：

1. 将 `tgo-platform` 正式接入 `snow-tgo-net`。
2. 将 `custom` 平台回调地址统一改为：
   - `http://ustarpay-kong/auth/v1/app/chat/callback`
3. 将 `custom` 平台配置中的 `platform_api_key` 同步为平台真实 `api_key`。

修复后已完成实测：

1. `tgo-platform -> /v1/messages/send` 返回 `200`
2. `nh_wallet.app_chat_messages` 已收到最新客服回复
3. App 聊天历史可查到客服回消息

### 6.4 测试服数据库结构落后

租户后台用户列表 / KYC 列表曾报错：

- `column k.operator_id does not exist`

当前已同步最近 SQL 修复：

- `2026-03-17_tenant_user_kyc_review.sql`
- `2026-03-17_tenant_user_country_code_nullable.sql`

### 6.5 测试服数据库访问策略

按最新安全要求，测试服数据库不再建议开放公网端口。

当前标准方案为：

1. PostgreSQL 仅绑定服务器本机回环地址：
   - `127.0.0.1:15432 -> 5432`
2. 本地开发或排障时，通过 SSH 隧道访问数据库。
3. 业务流量仍统一走 Kong，数据库不作为公网业务入口。

说明：

1. 这样不会影响 UStarPay 与 TGO 容器内部使用 `postgres:5432` 的业务链路。
2. 只会影响外部直接连数据库的方式。
3. 后续团队统一通过 SSH 隧道 + Navicat / DBeaver 访问测试服数据库。
