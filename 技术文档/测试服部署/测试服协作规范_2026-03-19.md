# 测试服协作规范
日期：2026-03-19  
适用范围：UStarPay + TGO 测试服协作、发布、数据库变更、配置变更

## 1. 目标

这份规范用于统一测试服协作方式，核心目标只有一条：

**任何人提交代码、发布测试服、执行 SQL，都不能影响测试服当前正在进行的体验与联调。**

当前测试服已经进入多人共用阶段，后续所有改动都要遵守：

1. 先看影响范围，再发布。
2. 先自测，再同步测试服。
3. 先最小改动，再最小重建。
4. 能不影响测试中的同事，就不要扩大操作面。

## 2. 默认发布范围

后续如果只说“发布测试服”或“同步测试服”，默认同步以下 6 套仓库：

1. `ustarpay-app`
2. `ustarpay-backend-go`
3. `ustarpay-db`
4. `ustarpay-frontend-admin`
5. `ustarpay-website`
6. `tgo`

其中数据库变更统一以：

- `D:\snow\ustarpay-db`

为唯一 SQL 提交入口。

## 3. 测试服发布原则

### 3.1 最高原则：不能影响测试服现网体验

发布时默认遵守：

1. 不做无关服务重启。
2. 不做整套全量重建，除非确认必要。
3. 不在同事集中体验时执行高风险操作。
4. 不直接修改测试服线上源码，统一通过 git 拉代码。
5. 发布后必须完成最小闭环验收。

### 3.2 以本地代码为准

后续测试服发布默认口径：

1. 以本地仓库已提交、已推送代码为准。
2. 测试服统一 `git pull` 同步。
3. 不允许“服务器改一版、本地又是一版”的双轨状态。

### 3.3 SQL 统一仓库管理

数据库变更必须遵守：

1. 所有新增 SQL 统一进入 `ustarpay-db` 仓库。
2. 文件名必须带日期，便于多人协作追踪。
3. 优先写成可重复执行，或在备注里明确说明不可重复执行。
4. 发布前先判断是否适合重复落库。
5. 发布后要做字段、表、索引、函数验收。

## 4. 同事提交建议

为了让测试服发布更稳定，建议各位同事按下面方式协作：

### 4.1 提交数据库变更时

1. 请统一提交到 `ustarpay-db`。
2. 不要把 SQL 藏在其他业务仓库、临时目录或聊天记录里。
3. 如果 SQL 依赖新函数、新表或新字段，请在提交说明里写清楚。
4. 如果 SQL 不能重复执行，请明确备注“不可重复执行”。

### 4.2 提交配置变更时

以下类型请在 commit message 或群里明确说明：

1. 端口变更
2. Docker / Compose 变更
3. Kong 路由变更
4. 环境变量变更
5. 回调地址变更
6. 鉴权方式变更
7. 数据库连接方式变更

原因：
这些内容会直接影响测试服发布，不提前说明，最容易出现“代码没问题，但环境挂了”。

### 4.3 不建议提交的内容

以下内容原则上不要提交到公共仓库：

1. `node_modules`
2. 本地日志文件
3. 本地构建产物
4. `.vite` 缓存
5. 本地临时调试脚本
6. 仅本机使用的环境覆盖文件
7. 本地数据库备份文件

### 4.4 本地调试配置建议

如果只是本地调试，请优先使用：

1. 本地 `.env`
2. 本地启动脚本
3. 本地测试数据

不要直接改公共测试服配置，避免影响其他同事联调。

## 5. 运维发布口径

后续测试服发布统一按标准 SOP 执行：

1. 先同步本地仓库
2. 先看最近 `git log`
3. 先同步 `ustarpay-db`
4. 按时间顺序执行新增 SQL
5. 按影响范围重建服务
6. 做页面、接口、聊天链路验收
7. 记录发布结果

如果发布过程中发现以下情况，会暂停继续发布并先排查：

1. 同一时间多人改了同一块配置
2. SQL 依赖关系不明确
3. 提交包含未说明的部署变更
4. 存在可能影响正在体验同事的高风险重启

## 6. 当前建议给团队的沟通口径

可以直接发给同事：

> 后续测试服发布会统一按标准 SOP 执行，优先保证测试服稳定，不影响大家当前体验。  
> 如果提交里涉及数据库、配置、Kong、Docker、端口、回调地址、环境变量等内容，请务必提前备注影响范围。  
> SQL 统一走 `ustarpay-db` 仓库提交，不要分散提交。  
> 本地运行产物、日志、缓存、临时脚本不要提交到公共仓库。  
> 如果只是本地调试，请尽量走本地配置，不要直接改公共部署配置。

## 7. 结论

测试服现在已经不是个人环境，而是多人共享环境。  
后续所有发布动作都要围绕两个关键词：

1. **稳定**
2. **可追踪**

这套流程跑顺后，后续平移到正式服也会更稳。
