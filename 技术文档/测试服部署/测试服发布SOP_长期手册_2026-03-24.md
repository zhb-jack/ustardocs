# 测试服发布 SOP 长期手册
更新时间：2026-03-24

## 1. 文档目标

这份手册只解决一件事：

在多人共用的测试服环境里，如何稳定、可追踪地完成代码发布、SQL 同步、服务重建和最小闭环验收。

这份 SOP 以后作为长期标准口径使用，不再依赖聊天记录、临时口头说明或个人记忆。

## 2. 适用范围

默认适用于 `D:\snow` 工作区内的测试服发布：

- `ustarpay-app`
- `ustarpay-backend-go`
- `ustarpay-db`
- `ustarpay-frontend-admin`
- `ustarpay-website`
- `ustarpay-ops`

如果本次需求涉及客服、IM、聊天或 TGO 联动，再额外纳入：

- `tgo`

## 3. 发布总原则

### 3.1 第一原则：稳定

测试服是共享环境，不是个人沙盒。发布时默认遵守：

1. 先判断影响范围，再动服务。
2. 先最小同步，再最小重建。
3. 能不影响同事正在体验的模块，就不要扩大操作面。
4. 不做无关服务重启。
5. 不做整套全量重建，除非确认必要。

### 3.2 第二原则：可追踪

发布必须可回溯，至少满足：

1. 本地代码已提交、已推送。
2. 测试服统一通过 `git pull` 同步，不直接在服务器长期改源码。
3. SQL 有独立脚本，有执行记录。
4. 发布后有明确的验收结果。

### 3.3 第三原则：代码、SQL、验证必须成套

测试服发布不是“把代码传上去”就算完成，标准动作必须包含：

1. 同步代码
2. 同步 SQL
3. 执行 SQL
4. 重建受影响服务
5. 验证页面、接口和关键业务闭环

少任何一步，都不算真正发布完成。

## 6. 标准发布前检查

正式登录测试服前，先在本地做 6 件事。

### 6.1 确认改动范围

明确这次涉及的是哪几个仓库、哪几个服务、是否有 SQL、是否有配置变更。

至少要回答这几个问题：

1. 改的是前端、后端、App、官网，还是多仓联动。
2. 是否新增或修改了 SQL。
3. 是否改了 Dockerfile、Compose、Kong、环境变量。
4. 是否涉及 TGO。

### 6.2 检查本地工作树

每个相关仓库都先看：

```powershell
git status --short
git log --oneline -5
```

### 6.3 本地先提交再发布

标准顺序：

```powershell
git add .
git commit -m "feat/fix: xxx"
git push origin <branch>
```

原则：

- 不要拿本地未提交的代码去发测试服。
- 不要让服务器和本地变成两套不同源码状态。

### 6.4 本地最小自测

至少做一轮和本次需求直接相关的最小闭环检查，比如：

- 页面能打开
- 关键接口返回正常
- 关键表单能提交
- 关键业务链路前后能走通

### 6.5 明确这次要执行的 SQL

先列出本次发布涉及的 SQL 文件，尤其是：

- 日期增量 SQL
- RBAC SQL
- 数据修复 SQL

### 6.6 判断是否属于高风险发布

以下任一情况，都要先额外停一下再发：

1. 变更涉及资金链路
2. 变更涉及公共鉴权
3. 变更涉及 Kong 路由
4. 变更涉及 Docker Compose
5. 变更涉及数据库结构
6. 变更涉及多人同时在测的公共模块

## 7. 登录测试服后的标准动作

### 7.1 先进入统一根目录

```bash
cd /opt/snow
```

### 7.2 先看服务器当前状态

进入每个相关仓库前，先检查：

```bash
git status --short
git branch --show-current
git rev-list --left-right --count HEAD...origin/<branch>
```

目的：

- 看工作树是否脏
- 看当前分支是否正确
- 看是否落后远端

### 7.3 如果工作树是脏的，先备份再处理

禁止直接：

- `git reset --hard`
- `git checkout -- .`

正确做法：

1. 先导出差异备份
2. 再 `stash`
3. 再拉代码
4. 最后按需 `stash pop`

示例：

```bash
mkdir -p /opt/snow/backups/git-release-$(date +%Y%m%d)
git diff > /opt/snow/backups/git-release-$(date +%Y%m%d)/repo-name-before-pull.patch
git stash push -u -m "release-backup-$(date +%Y%m%d-%H%M%S)"
git pull --ff-only origin <branch>
git stash list
```

说明：

- 如果 `stash pop` 会造成冲突，先停下来处理，不要硬上。
- 测试服上出现脏工作树，优先保留现场，不要粗暴覆盖。

## 9. SQL 发布标准步骤

### 9.1 原则

数据库变更必须遵守：

1. 所有 SQL 先进入 `ustarpay-db`
2. 文件名带日期
3. 先确认执行顺序，再执行
4. 执行前先做备份
5. 执行后做结构或数据验收

### 9.2 执行前备份

至少备份这次会受影响的表或相关结构。

通用示例：

```bash
mkdir -p /opt/snow/backups/git-release-$(date +%Y%m%d)
docker exec ustarpay-postgres pg_dump -U postgres -d nh_wallet -t tenant_configs > /opt/snow/backups/git-release-$(date +%Y%m%d)/tenant_configs.sql
```

如果影响多张表，就分别备份或直接导出相关 schema。

### 9.3 执行 SQL

标准示例：

```bash
docker exec -i ustarpay-postgres psql -U postgres -d nh_wallet < /opt/snow/ustarpay-db/2026-03-24_xxx.sql
```

如果这次有多份 SQL，按依赖顺序执行，不要乱序。

常见顺序建议：

1. 函数或基础结构
2. 主业务表
3. 索引、约束
4. RBAC 或初始化数据
5. 数据修复

### 9.4 执行后验收

至少确认以下一项或多项：

```bash
docker exec -it ustarpay-postgres psql -U postgres -d nh_wallet
```

进入后检查：

- 表是否存在
- 字段是否存在
- 索引是否存在
- 配置数据是否落库
- RBAC 权限是否落库

说明：

- SQL 同步到仓库不代表发布完成。
- 只有“脚本在库里 + 已执行到测试服 + 已验证”才算完成。

## 10. 服务重建标准步骤

统一入口：

```bash
cd /opt/snow/ustarpay-ops/deploy/staging
```

编排文件：

```bash
docker compose -f docker-compose.ustarpay.yml
```

### 10.1 服务名对照

| 改动范围 | 对应服务 |
| --- | --- |
| `ustarpay-app` | `app-web` |
| `ustarpay-frontend-admin` 平台后台 | `admin-platform` |
| `ustarpay-frontend-admin` 租户后台 | `tenant-portal` |
| `ustarpay-website` | `website` |
| `auth` 相关后端 | `auth-service` |
| App 业务、钱包、卡、充值提现 | `tenant-api` |
| 平台后台 API | `admin-service` |
| 租户后台 API | `tenant-admin-service` |
| 回调、异步处理 | `webhook-service` |
| Kong 或路由配置 | `kong` |

### 10.2 标准重建命令

单服务：

```bash
docker compose -f docker-compose.ustarpay.yml up -d --build tenant-api
```

多服务：

```bash
docker compose -f docker-compose.ustarpay.yml up -d --build tenant-api tenant-admin-service webhook-service
```

前端服务：

```bash
docker compose -f docker-compose.ustarpay.yml up -d --build app-web tenant-portal admin-platform website
```

Kong：

```bash
docker compose -f docker-compose.ustarpay.yml up -d --build kong
```

### 10.3 什么情况下需要重建 `kong`

只有改了下面这些内容，才考虑动 `kong`：

1. `docker-compose.ustarpay.yml`
2. `kong/kong.yml`
3. Kong 相关环境变量
4. 前端挂载路径或代理规则

### 10.4 如果 compose 报服务名识别异常

先在当前目录执行：

```bash
docker compose -f docker-compose.ustarpay.yml config --services
```

先确认：

1. 当前目录是否在 `/opt/snow/ustarpay-ops/deploy/staging`
2. 编排文件名是否正确
3. 服务名是否与 compose 中一致

不要在目录不对、文件不对的情况下盲重试。

## 11. 发布后最小闭环验收

### 11.1 页面验收

按本次影响范围访问相关页面，至少确认返回正常页面而不是白屏或 502。

常见入口包括：

- `/admin/`
- `/tenant/`
- `/app/`
- `/website/`

### 11.2 接口验收

至少做一轮真实接口检查，优先验证本次改动相关链路。

例如：

- 后台登录
- App 登录
- 钱包资产接口
- 资金流水接口
- KYC 列表
- 邀请返佣配置接口
- Binance Pay 下单接口

### 11.3 业务验收

如果这次改的是业务功能，不要只看 200，还要走一轮最小业务闭环。

例如：

- 钱包：资产 -> 交易记录 -> 地址簿 -> 提现预检
- 邀请返佣：菜单 -> 权限 -> 配置 -> 事件 -> 结算
- Binance Pay：创建订单 -> 返回支付参数 -> 刷新订单状态
- 卡业务：预检 -> 申请 -> 列表 -> 状态变更

### 11.4 容器和日志检查

```bash
docker ps
docker logs --tail 100 ustarpay-tenant-api
docker logs --tail 100 ustarpay-tenant-admin-service
docker logs --tail 100 ustarpay-kong
```

如果这次涉及其他服务，再补对应日志。

## 12. 默认发布范围口径

以后如果只说“发布测试服”，默认至少检查并同步下面这 6 套仓库是否需要更新：

1. `ustarpay-app`
2. `ustarpay-backend-go`
3. `ustarpay-db`
4. `ustarpay-frontend-admin`
5. `ustarpay-website`
6. `ustarpay-ops`

如果本次需求涉及客服、聊天、TGO，再额外纳入：

7. `tgo`

说明：

- `ustarpay-frontend-admin` 不能漏
- `ustarpay-db` 不能只同步脚本不执行
- `ustarpay-ops` 也要检查，尤其是 Dockerfile、compose、Kong 配置变更

## 13. 高风险场景额外要求

以下场景发布时，必须额外做影响评估和验收记录：

1. 资金链路
2. KYC 审核链路
3. 登录鉴权
4. Kong 路由
5. 多仓联动
6. SQL 结构变更

额外要求：

1. 发布前明确影响面
2. 发布后至少走一轮真实业务闭环
3. 必要时记录回滚或补救方案

## 15. 一句话结论

以后测试服发布的标准动作就是：

本地提交并推送 -> 测试服安全同步代码 -> 备份并执行 SQL -> 最小范围重建服务 -> 最小闭环验收 -> 留发布记录。
