# UStarPay 邀请返现需求梳理

## 1. 目标与边界

本文用于沉淀 UStarPay “邀请返现”需求在当前系统中的落点、缺口和最小改动方案，作为后续设计、开发、测试、发布的统一依据。

当前执行原则：

1. 不改主架构。
2. 优先复用现有用户体系、KYC、卡业务、资金账户、资金流水、租户配置、调度框架。
3. 先做最小闭环，再考虑等级、排行榜、复杂营销扩展。
4. 后台模块命名与 UI 风格必须与现有管理后台保持统一，不单独造一套视觉。

## 2. 业务规则基线

### 2.1 已确认的业务规则

1. 邀请人需要完成 KYC，并生成专属邀请链接或邀请码。
2. 被邀请人需要通过邀请链接或邀请码注册。
3. 被邀请人完成 KYC，并完成开卡相关动作后，才算成功邀请。
4. 返佣以 USDT 结算，进入 UStarPay 钱包资金账户。
5. 开卡佣金为一次性返佣。
6. 消费佣金为长期返佣，按后台配置 T+N 结算。
7. 当前最多只往上追两层父子关系。
8. 后台可配置一级、二级分佣比例。
9. 普通用户走固定比例，专属级用户（KOL / 合作方）支持单用户差异化比例。

### 2.2 当前存在的业务歧义

以下两点必须在正式开发前定口径：

1. “消费佣金”到底按“划转至 VISA 卡金额”返，还是按“卡消费结算金额”返。
2. “成功邀请”到底以“完成 KYC + 成功购卡”为准，还是必须“完成 KYC + 购卡成功 + 激活成功”才触发开卡佣金。

推荐默认口径：

1. 开卡佣金按“完成 KYC + 成功购卡”触发，不再额外等待激活。
2. 消费佣金按“卡消费结算金额”触发，走 T+N 结算。

推荐原因：

1. 当前代码里“购卡成功”和“激活成功”是两个独立节点，若把激活也作为返佣前置条件，会拉长业务链路并增加争议单。
2. 当前系统已有稳定的卡消费结算事件，但尚未落地真实的 `transfer_to_card` 业务实现。

## 3. 现有能力映射

### 3.1 邀请关系已存在

当前系统注册时已支持邀请码绑定：

1. `tenant_user.invited_uid` 记录直接上级。
2. `tenant_user_invite_binding` 记录绑定快照与来源。
3. App 端邀请链接、邀请码、邀请人、邀请列表接口已存在。

关键落点：

1. `D:\snow\ustarpay-backend-go\internal\domains\auth\service\auth.go`
2. `D:\snow\ustarpay-backend-go\internal\infrastructure\database\auth\repository\auth.go`
3. `D:\snow\ustarpay-backend-go\internal\domains\tenant-user\models.go`
4. `D:\snow\ustarpay-db\tenant\tenant_user.sql`

说明：

1. 当前是单父链结构。
2. 两层分佣可以通过“当前用户 -> 直接上级 -> 上级的上级”实现。
3. 现有 `idx_tenant_user_invited_uid` 索引已具备基础查询能力。

### 3.2 KYC 体系已存在

当前系统已有 L1 / L2 KYC 状态同步：

1. L1 审核通过后，会更新 `tenant_user.is_kyc_internal = 1`。
2. L2 审核通过后，会更新 `tenant_user.is_kyc_genuine = 1`。
3. App 端已能读取当前用户 KYC 状态。

关键落点：

1. `D:\snow\ustarpay-backend-go\internal\adapters\http\tenant-admin\handler\tenant_user.go`
2. `D:\snow\ustarpay-backend-go\internal\domains\auth\service\auth.go`
3. `D:\snow\ustarpay-backend-go\internal\infrastructure\database\auth\repository\auth.go`

建议：

1. 成功邀请中的“KYC 完成”第一版直接以 L1 通过为准，即 `is_kyc_internal = 1`。
2. 若后续业务明确要求 L2 也必须通过，再升规则，不要第一版直接抬高门槛。

### 3.3 开卡、激活、消费链路已存在

当前卡业务已具备关键事件节点：

1. 实时开卡成功后，会确认开卡费用扣减，并创建卡记录。
2. 预制实体卡申请单存在独立审核流，审核通过并绑卡后状态变为 `1`。
3. 实体卡激活接口已存在。
4. 卡状态 Webhook 已支持 `ACTIVATED` 状态同步。
5. 卡消费授权与结算 Webhook 已存在。

关键落点：

1. `D:\snow\ustarpay-backend-go\internal\domains\card\service\card_service.go`
2. `D:\snow\ustarpay-backend-go\internal\domains\card\service\physical_card_admin.go`
3. `D:\snow\ustarpay-backend-go\internal\domains\card\service\webhook_service.go`
4. `D:\snow\ustarpay-backend-go\internal\domains\card\models.go`

建议的事件落点：

1. 虚拟卡 / 实时实体卡开卡佣金：挂在实时开卡成功后。
2. 预制实体卡开卡佣金：挂在申请审核通过并绑卡后。
3. 激活状态：作为展示字段和后续风控辅助，不建议作为第一版返佣前置条件。
4. 消费佣金：挂在卡消费结算成功事件。

### 3.4 钱包账户与资金流水已存在

当前资金体系已满足返佣入账基础条件：

1. 用户资金账户表已存在。
2. 统一资金流水表 `tenant_fund_flow_records` 已存在。
3. `invite_rebate` 已经是预留好的流水类型。

关键落点：

1. `D:\snow\ustarpay-backend-go\internal\domains\tenant-user\models.go`
2. `D:\snow\ustarpay-backend-go\internal\domains\fundflow\models.go`
3. `D:\snow\ustarpay-backend-go\internal\domains\payment\service\service.go`
4. `D:\snow\ustarpay-backend-go\internal\domains\card\service\card_service.go`
5. `D:\snow\ustarpay-backend-go\internal\domains\card\service\webhook_service.go`

说明：

1. 开卡费扣减已经复用资金流水。
2. 邀请返现可以沿用同一套入账模型，不需要另造钱包体系。

### 3.5 配置与调度能力已存在

当前系统已有租户配置和 cron 调度能力：

1. `tenant_config` 可承接返佣基础规则。
2. collection 模块已有调度框架，可复用 T+N 结算任务思路。

关键落点：

1. `D:\snow\ustarpay-backend-go\internal\domains\tenant\repository\config.go`
2. `D:\snow\ustarpay-backend-go\internal\domains\tenant\service\config.go`
3. `D:\snow\ustarpay-backend-go\pkg\shared\enum.go`
4. `D:\snow\ustarpay-backend-go\internal\domains\collection\service\scheduler.go`

建议：

1. 第一版新增配置键 `invite_rebate`，配置值先存 JSON。
2. 普通用户基础比例、一级 / 二级比例、T+N 参数先放租户配置。
3. 专属级用户差异化比例用独立档案表承接。

## 4. 前后端现状与缺口

### 4.1 App 端现状

当前 `referral.tsx` 已接真实邀请接口，但仍有大量演示数据：

1. 邀请链接、邀请码、邀请人数、邀请列表、邀请人信息是真实接口。
2. analytics / earnings / tier / leaderboard 等内容目前仍是前端静态占位。

关键落点：

1. `D:\snow\ustarpay-app\app\referral.tsx`
2. `D:\snow\ustarpay-app\features\auth\types.ts`

结论：

1. App 端第一版不要继续强化演示型数据。
2. 先把返佣总额、待结算、已结算、返佣规则说明替换成真实接口字段。
3. 排行榜、等级路线图不建议第一版上线。

### 4.2 后台现状

当前后台已有“邀请关系”页面，但没有返佣台账能力：

1. 已有邀请关系列表、邀请树、单用户邀请详情。
2. 已有资金流水页，并已识别 `invite_rebate` 类型标签。
3. 当前没有返佣配置、返佣事件、返佣结算管理页。

关键落点：

1. `D:\snow\ustarpay-frontend-admin\tenant-portal\src\components\Layout\MainLayout.tsx`
2. `D:\snow\ustarpay-frontend-admin\tenant-portal\src\pages\users\InvitationListPage.tsx`
3. `D:\snow\ustarpay-frontend-admin\tenant-portal\src\pages\transactions\FundFlowListPage.tsx`

## 5. 后台命名与 UI 统一规范

### 5.1 菜单命名建议

后台建议统一命名为：

1. 一级能力名称：`邀请返现`
2. 子页面一：`邀请关系`
3. 子页面二：`返佣配置`
4. 子页面三：`返佣记录`
5. 子页面四：`结算记录`

说明：

1. “邀请关系”保留现有关系管理含义。
2. “返现”比“返佣”更贴近业务对外表达，但后台数据明细页可用“返佣记录”“结算记录”体现财务语义。
3. 后台文案中弱化“租户”“总后台”等词，面向业务方统一使用“平台”“后台”“技术维护后台”等表述。

### 5.2 UI 风格规范

后续页面必须遵守：

1. 复用现有后台布局、卡片、表格、筛选器、抽屉、详情区组件风格。
2. 不做大面积英雄图、夸张图表、超大标题版式。
3. 优先延续现有 `InvitationListPage` 与 `FundFlowListPage` 的信息密度与节奏。
4. 返佣页以“筛选 + 列表 + 详情 / 抽屉”为主，不做与现有后台割裂的营销视觉页面。

## 6. 推荐的最小改动方案

### 6.1 复用现有能力

第一版直接复用：

1. `tenant_user.invited_uid`
2. `tenant_user_invite_binding`
3. `tenant_user_accounts`
4. `tenant_fund_flow_records`
5. `tenant_config`
6. collection scheduler 思路

### 6.2 建议新增的数据表

第一版建议新增 3 张表：

1. `tenant_invite_rebate_profiles`
2. `tenant_invite_rebate_events`
3. `tenant_invite_rebate_settlements`

建议职责：

1. `tenant_invite_rebate_profiles`
   - 记录用户返佣档案
   - 是否普通 / 专属级
   - 单用户开卡佣金比例
   - 单用户消费佣金比例
   - 生效状态
2. `tenant_invite_rebate_events`
   - 记录返佣来源事件
   - 事件类型：开卡返佣 / 消费返佣
   - 被邀请人、一级上级、二级上级快照
   - 原始业务单号、卡单号、交易单号
   - 比例快照、金额快照、状态
   - 是否已生成资金流水
3. `tenant_invite_rebate_settlements`
   - 记录 T+N 结算批次 / 单据
   - 结算日期、规则快照、执行结果、失败原因

设计要求：

1. 返佣事件表必须固化一级、二级上级快照，避免后续用户关系变化影响追溯。
2. 返佣事件表必须设计幂等键，防止重复结算。
3. 开卡返佣可直接标记为可结算或已结算。
4. 消费返佣先记事件，再由结算任务入账。

### 6.3 建议新增的租户配置

建议在 `tenant_config` 中新增 `invite_rebate` 配置键，JSON 至少包含：

1. 普通用户开卡返佣比例
2. 普通用户消费返佣比例
3. 一级上级分佣比例
4. 二级上级分佣比例
5. 消费返佣结算周期 T+N
6. 是否启用开卡返佣
7. 是否启用消费返佣
8. 开卡返佣是否实时入账

### 6.4 事件触发建议

建议按以下节点落事件：

1. 注册绑定：沿用现有逻辑，不额外记返佣事件。
2. L1 KYC 通过：作为“是否具备邀请资格 / 是否具备被邀请人有效资格”的判断条件。
3. 实时开卡成功：生成一次性开卡返佣事件。
4. 预制实体卡审核通过并绑卡：生成一次性开卡返佣事件。
5. 卡消费结算成功：生成消费返佣事件。
6. T+N 定时任务：扫描到期消费返佣事件并批量入账。

## 7. 实现顺序建议

建议按以下顺序推进：

1. 先补 SQL 脚本到 `D:\snow\ustarpay-db`
2. 后端补返佣配置读取、事件落库、结算任务、钱包入账
3. 后台补菜单、配置页、返佣记录页、结算记录页
4. App 邀请页替换真实返现数据
5. 最后补自测与测试服发布

## 8. 自测范围建议

第一版最小自测要覆盖：

1. 邀请注册绑定是否正常写入关系
2. 一级 / 二级上级追溯是否正确
3. 未完成 KYC 的被邀请人是否不会触发返佣
4. 虚拟卡开卡返佣是否只触发一次
5. 预制实体卡审核通过返佣是否只触发一次
6. 卡消费结算返佣是否按 T+N 入账
7. 同一业务单重复 webhook 是否不会重复返佣
8. 返佣资金流水是否落到 `invite_rebate`
9. 后台配置修改后是否只影响新事件，不污染旧事件快照

## 9. 当前结论

当前系统已经具备做“邀请返现”第一版的基础能力，不需要改主架构。

真正需要补的是：

1. 返佣事件台账
2. 专属级用户返佣档案
3. T+N 结算任务
4. 后台返佣管理页面
5. App 端真实返现数据接口

当前唯一必须先拍板的业务问题：

1. 消费佣金口径
2. 开卡佣金是否要求“激活”作为前置条件

如果没有新的业务纠偏，后续默认按本文推荐口径推进。
