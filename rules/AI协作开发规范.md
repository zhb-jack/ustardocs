# AI 协作开发规范

## 1. 目的

本规范用于指导 AI 助手进行代码开发、修改、排查与文档更新，确保所有操作：

1. 安全
2. 可控
3. 可维护
4. 符合当前项目既定架构与协作方式

适用范围：

1. Cursor
2. Codex
3. Claude
4. 其他参与 `D:\snow` 工作区开发与文档维护的 AI 助手

## 2. 使用建议

在向 AI 助手提出复杂任务时，建议明确补充：

```text
请严格按照 D:\snow\docs\rules\AI协作开发规范.md 执行。
```

如需强调测试服、SQL、文档归档规则，可继续补充：

```text
请同时遵守 D:\snow\docs\rules\README.md 中的长期规则与 SOP 约定。
```

## 3. 规则优先级

当多条规则冲突时，按以下优先级决策：

1. 系统安全性
2. 架构一致性
3. 代码可维护性
4. 功能实现

补充说明：

1. 如果所使用的 AI 平台本身存在更高优先级的系统安全规则，以平台安全规则为上限。
2. 在不违反平台规则的前提下，尽量完整执行本规范。

## 4. 核心协作原则

### 4.1 语言与交互

1. 默认使用中文回复。
2. 如用户明确指定称呼，则按用户要求称呼。
3. 在 Cursor 等可定制场景下，如未另行指定，可使用“张哥：”作为称呼前缀。
4. 回复要直接、清楚、可执行，避免空泛表达。

### 4.2 任务拆解

1. 面对复杂需求，先拆解为可执行的小任务。
2. 优先给出清晰步骤、边界与落点，再实施。
3. 复杂任务应分阶段推进，避免一次性扩散改动面。

### 4.3 谨慎询问

遇到以下情况时，必须先澄清再实施：

1. 需求本身存在歧义
2. 架构边界不清晰
3. 会影响多个模块或公共契约
4. 有明显高风险或不可逆操作

禁止在关键前提不明时擅自决定或直接落地。

## 5. 安全与风险控制

### 5.1 信息保护

1. 不得在代码、注释、日志、文档或回复中泄露敏感信息。
2. 敏感信息包括但不限于：
   - 密钥
   - 口令
   - Token
   - 内部 IP
   - 生产数据库连接信息
   - 私有证书
3. 如必须提及敏感配置，只描述用途与位置，不输出真实值。

### 5.2 数据库变更

1. 所有涉及数据库结构变更的操作，必须优先生成 SQL 变更脚本。
2. 所有涉及重要数据修正的操作，必须优先生成 SQL 脚本并提供审核。
3. 禁止直接跳过脚本，在数据库中临时手改后不留痕。
4. 结合当前项目约定，新增 SQL 一律放到：
   - `D:\snow\ustarpay-db`
5. SQL 文件名必须带年月日。
6. 如 SQL 不可重复执行，必须在文件或说明中明确标注。

### 5.3 高风险操作控制

在执行以下操作前，必须先说明影响范围，并提醒或建议备份：

1. 批量重命名
2. 删除文件
3. 移动文件
4. 重构公共 API
5. 修改共享配置
6. 大范围替换
7. 数据迁移

### 5.4 影响评估

高风险操作实施前，至少要分析以下内容：

1. 影响哪些模块
2. 是否影响现有接口或页面
3. 是否影响测试服或多人联调
4. 是否存在兼容性风险
5. 是否需要回滚预案

## 6. 架构与代码质量

### 6.1 架构遵从

1. 严格遵守现有项目架构、目录分层、命名方式和代码风格。
2. 优先融入当前体系，不另起一套新架构。
3. 如果现有架构不清晰，应先梳理依赖关系，再修改。

### 6.2 变更最小化

1. 每次修改应围绕一个明确目标展开。
2. 避免把无关修复、风格调整、重构混在同一轮里。
3. 优先采用最小改动路径。
4. 功能增强时，必须保证不影响原有功能正常使用。

### 6.3 设计原则

设计时：

1. 用第一性原理追问问题本质。
2. 避免为未来假设过度设计。

实现时：

1. 遵循 KISS 原则。
2. 遵循 SOLID 原则。
3. 复杂逻辑优先拆分，而不是在单点持续堆积。

### 6.4 复用优先

1. 添加新功能前，必须先检查现有代码库是否已有可复用模块。
2. 优先复用现有：
   - service
   - repository
   - 页面组件
   - 工具函数
   - 类型定义
   - 现有 SQL 结构
3. 严禁无必要重复造轮子。

### 6.5 依赖管理

1. 不引入不必要的第三方依赖。
2. 如必须新增依赖，需要说明：
   - 为什么必须引入
   - 是否已有可替代方案
   - 维护成本和后续影响

### 6.6 代码可读性

1. 代码必须清晰可读。
2. 复杂逻辑可加简短注释，重点解释“为什么这么做”。
3. 不写低信息量注释。
4. 保持 import、函数命名、错误处理风格一致。

### 6.7 函数复杂度

1. 鼓励函数精简。
2. 单个函数或方法建议不超过 80 行。
3. 避免高圈复杂度、深层嵌套、隐式副作用过多。

## 7. 开发执行规范

### 7.1 功能增强

在已有功能上新增能力时，必须满足：

1. 不影响原有功能
2. 不引入计划外额外功能
3. 不追加无关配置
4. 不扩大不必要改动范围

### 7.2 代码优化

生成或修改代码后，应主动做基础优化：

1. 清理未使用 import
2. 合并明显重复逻辑
3. 避免死代码
4. 保持命名统一

如发现明显性能风险，应主动提示，例如：

1. 复杂 SQL
2. 深层嵌套循环
3. 重复请求
4. 无索引查询
5. 大对象重复构造

### 7.3 异常处理

关键功能必须有基本异常处理，特别是：

1. 外部调用
2. 文件 IO
3. 数据转换
4. 数据落库
5. 定时任务
6. 远程接口请求

目标是避免程序非正常崩溃，并让错误具备可定位性。

### 7.4 逻辑自检

任何代码实现完成后，必须做基础自检，至少包括：

1. 边界条件
2. 空值处理
3. 错误分支
4. 数据流是否闭环
5. 返回值是否一致
6. 是否引入明显回归风险

## 8. 场景化细则

### 8.1 接口与契约

修改以下内容时，必须同步评估兼容性并更新相关文档：

1. 对外 API
2. 返回数据结构
3. 请求参数
4. 配置文件格式
5. 前后端契约

需要明确说明：

1. 是破坏性变更还是向后兼容
2. 受影响方有哪些
3. 需要哪些联调或回归验证

### 8.2 测试要求

1. 新增功能应补充相应测试用例。
2. 修改已有逻辑后，应确保相关现有测试可通过。
3. 如果当前仓库没有现成测试体系，也至少要完成最小闭环验证并记录验证方式。

### 8.3 文档要求

1. 核心公共代码、复杂算法、重要接口需要补充文档注释。
2. 如果某次修改已经形成可复用 SOP、长期规则或联调口径，要同步更新：
   - `D:\snow\docs\rules\README.md`
   - 当前规范文件
   - 对应业务文档

### 8.4 文档归档要求

1. 技术对接类文档优先放到：
   - `D:\snow\docs\接口文档对接`
2. 长期规则与总入口统一放到：
   - `D:\snow\docs\rules`
3. 业务专项的需求梳理、实现方案、联调记录、阶段性过程文档统一放到：
   - `D:\snow\docs\技术文档-工作日志\<专项目录>`
4. 这类业务专项文档必须单独建目录，不直接散放在 `技术文档-工作日志` 根目录。
5. 不要把技术对接文档默认放到：
   - `D:\snow\docs\需求文档`

## 9. 项目专项补充规则

### 9.1 主工作区

1. 当前默认主工作区为：`D:\snow`
2. 后续如果未特别说明，默认以 `D:\snow` 内仓库为主进行分析、修改与文档维护。

### 9.2 测试服专项规则

1. 测试服是多人共享环境。
2. 发布或修复时优先保证稳定，不打断其他同事联调。
3. 默认先本地自测，再同步测试服。
4. 测试服默认通过 `git pull` 同步，不直接在服务器改源码。
5. 默认只重建受影响服务，不做全量重建。

### 9.3 验收专项规则

后续默认优先做“真实可演示、可汇报”的验证：

1. 浏览器级真实访问
2. 接口级真实返回
3. 数据库或日志对账
4. 能做完整链路，就不只做静态代码判断

## 10. 维护机制

以下内容一旦新增、变化或稳定下来，必须同步更新本规范：

1. 长期有效的 AI 协作规则
2. 默认 SQL 规则
3. 默认文档归档规则
4. 默认测试服发布口径
5. 重要的验证标准
6. 与多人协作相关的新约束

## 11. 总结

本规范的目标不是限制开发速度，而是让 AI 助手在 `D:\snow` 工作区内做到：

1. 不乱改
2. 不越界
3. 不丢文档
4. 不破坏共享环境
5. 修改可追踪，结果可验收
