# 代码治理补充规则

## 1. 目的

补充约束 `D:\snow` 工作区内的代码文件规模，避免单文件持续膨胀后带来的维护成本、联调风险和回归难度。

## 2. 默认规则

1. 单个代码文件应尽量控制在 `500` 行以内。
2. 如确因框架约束、生成代码、聚合导出文件等原因暂时超过 `500` 行，必须先判断是否可以拆分：
   - 子组件拆分
   - hooks / service 拆分
   - types / constants 拆分
   - domain logic 拆分
3. 新增需求时，不应继续向已经明显超长的文件持续堆叠逻辑。

## 3. 执行口径

1. 该规则是默认治理目标，不要求为了压行数做无意义拆分。
2. 拆分优先保证：
   - 架构一致
   - 职责清晰
   - 调用关系稳定
   - 不影响现有功能
3. 如当前阶段为了业务交付需要暂时保留超长文件，应在同轮说明中明确：
   - 为什么本轮不拆
   - 后续准备如何拆

## 4. 优先级

当“快速交付”和“文件拆分”发生冲突时，按以下顺序决策：

1. 资金安全与业务正确性
2. 架构一致性
3. 可维护性
4. 文件行数治理

## 5. 适用范围

1. `ustarpay-app`
2. `ustarpay-backend-go`
3. `ustarpay-frontend-admin`
4. `ustarpay-website`
5. 其他后续纳入 `D:\snow` 主工作区协作的代码仓库
