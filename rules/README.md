# Snow 工作规则与 SOP 总览

## 1. 作用

这份文档是 `D:\snow` 的长期规则总入口，用来统一记录：

1. 跨仓库都适用的固定规则
2. 测试服发布与联调 SOP
3. 文档归档约定
4. 后续需要持续维护的协作口径

说明：

1. 细节型、阶段型、带日期的联调文档，仍然可以保留在原业务目录。
2. 一旦某个流程已经稳定、会反复复用，就要同步更新到这份总览。

## 2. 命名约定

当前目录命名采用：

- 目录：`rules`
- 主入口文件：`README.md`
- AI 详细规则文件：`AI协作开发规范.md`

约定说明：

1. 目录用 `rules` 比 `rule` 更规范，因为这里存放的是一组规则，而不是单条规则。
2. 如果后续拆分子文档，也继续放在 `D:\snow\docs\rules` 下。
3. 如果只是这一份总入口，保留 `README.md` 最稳，进入目录就能直接看到。

## 5. SQL 规则

1. 所有新增 SQL，统一放到：`D:\snow\ustarpay-db`
2. SQL 文件名必须带年月日，便于多人协作追踪
3. 不要把 SQL 散落在其他业务仓库、临时目录、聊天记录或测试脚本里
4. 如果 SQL 不可重复执行，需要在文档或备注里明确标注
5. 发布后需要做字段、表、索引、函数层面的基本验收

## 6. 测试服发布规则

### 6.1 基本原则

1. 测试服是多人共享环境，优先保证稳定
2. 发布时先看影响范围，再做最小改动
3. 不做无关服务重启
4. 不做整套全量重建，除非确认必要
5. 不直接在测试服改线上源码，统一通过 `git pull` 同步
6. 发布后必须做最小闭环验收

### 6.2 默认发布范围

如果只说“发布测试服”或“同步测试服”，默认同步以下 6 套仓库：

1. `ustarpay-app`
2. `ustarpay-backend-go`
3. `ustarpay-db`
4. `ustarpay-frontend-admin`
5. `ustarpay-website`
6. `tgo`

补充说明：

1. `ustarpay-ops` 只有在部署资产变更时再一起同步
2. 数据库变更统一以 `ustarpay-db` 为唯一 SQL 提交入口

### 6.3 默认分支

1. `ustarpay-app`：`dev`
2. `ustarpay-backend-go`：`master`
3. `ustarpay-db`：`master`
4. `ustarpay-frontend-admin`：`master`
5. `ustarpay-website`：`dev`
6. `ustarpay-ops`：`master`
7. `tgo`：`snow`

### 6.4 默认发布顺序

1. 本地开发完成
2. 本地自测通过
3. `git add` / `git commit` / `git push`
4. 测试服进入对应目录执行 `git pull --ff-only`
5. 只重建受影响服务
6. 做页面、接口、聊天链路验收
7. 记录发布结果

## 10. 验收口径

后续默认优先做“真实可演示、可汇报”的验收，不只停留在代码层。

默认优先级：

1. 浏览器级真实访问
2. 接口级真实返回
3. 数据库或日志级结果对账
4. 能做完整链路，就不只做单点验证

## 11. 实施偏好

1. 优先走最小改动路径
2. 优先做隔离，不随便碰无关模块
3. 能临时装依赖验证的，尽量不污染业务仓库
4. 共享环境下优先稳定和可追踪
5. 修复后优先给出真实结果，不只给理论分析

## 12. 维护约定

从现在开始，后续新增或变更以下内容时，需要同步更新本文件：

1. 长期有效的固定规则
2. 默认发布范围或默认分支
3. 测试服服务器、入口、部署方式
4. SQL 提交规则
5. 文档归档规则
6. 本地启动总入口
7. 已经稳定下来的 SOP
8. AI 助手执行口径的长期变化

补充要求：

1. 如果变化属于 AI 协作方式本身，也要同步更新 [AI协作开发规范.md](D:/snow/docs/rules/AI协作开发规范.md)
2. 如果只是阶段性实验，不要直接写进长期规则
3. 如果已经形成稳定 SOP，要尽快从业务文档回收进 `rules` 目录
