# UstarPay 项目现状与需求对比报告

基于对 `F:\snow\ustarpay-app` 现有代码结构的扫描，结合《Ustar 全角色资金业务流程》与《U卡产品技术架构设计&业务流程设计文档》的对比，以下是目前的**已实现模块**分析以及**缺失功能**的盘点：

## 1. 当前项目现状 (F:\snow\ustarpay-app)

### 技术栈吻合度
现有代码的前端技术栈与架构文档的规划基本**完全一致**：
*   **框架**：使用了 React Native (基于 Expo 框架，`expo-router`)。
*   **UI/样式**：配置了 `tailwindcss` 和 `nativewind`。
*   **状态与数据交互**：引入了 `zustand` (状态管理) 和 `@tanstack/react-query` (数据缓存请求)。
*   **其他关键库**：包括 `@stripe/stripe-react-native` (可能用于处理法币/银行卡相关)、`@react-native-async-storage/async-storage` (本地存储) 等。

### 已有业务骨架 (基于路由文件推测)
通过 `app/` 和 `app/(tabs)/`、`app/card/` 目录结构，目前项目已经搭建了以下页面的入口和初步界面：
*   **资产与核心交易**：`deposit.tsx` (充值)、`withdraw.tsx` (提现)、`swap.tsx` (闪兑/转换)、`send.tsx` (发送)、`receive.tsx` (接收)。
*   **U卡系统 (Card 模块)**：`apply.tsx` (申请开卡)、`topup.tsx` (卡面充值)、`details.tsx` (卡详情)、`freeze.tsx` (冻结持卡)、`limit.tsx` (限额设定)、`statements.tsx` (账单)、`history.tsx` (历史记录)。
*   **其他功能页面**：`address-book.tsx` (地址簿)、`merchant.tsx` (商户)、`onboarding.tsx` (引导页)、`profile.tsx` (个人中心)、`referral.tsx` (推荐/邀请)、`three-ds.tsx` (3DS 验证)、`notifications.tsx` (通知)。

## 2. 缺失与未完善的功能 (Missing Features)
虽然项目已经有了丰富的页面入口，但对比详细的需求流程，当前系统在**具体业务逻辑集成、合规与后端交互**等方面，仍然存在大量需要开发或对接的缺失功能：

### A. 账户与认证体系 (Account & KYC)
*   **未见完整的 KYC (实名认证) 提交流程**：文档强调需要遵从 AML (反洗钱) 规范。目前虽然有 `profile.tsx` 和 `onboarding.tsx` 页面，但未发现如上传身份证明、活体检测或对接第三方 KYC 服务的具体页面组件(如 Jumio/Sumsub 等)，这对金融级应用至关重要。
*   **登录与安全验证 (Security)**：未完全体现 JWT Auth 相关的拦截器设计；生物识别 (`expo-local-authentication` 有引入但需确认是否贯穿了高危操作如支付、提现过程)；复杂的密码/PIN重置流程。

### B. 核心资交流程闭环 (Core Funds Flow)
*   **StraitsX 及法币通道回调对接**：虽然有充提页面，但文档明确提到了 `StraitsX` 账户体系（支持实体商家结算）。前端目前缺少与之明确对应的商户入驻、结算绑定相关的特殊信息录入和状态展示（可能 `merchant.tsx` 还需要深入拓展）。
*   **手续费与卡费扣减逻辑展示**：虚拟卡与实体卡（$10与$100开卡费）的费用扣除、换汇手续费 (USDT->法币) 的实时试算 (Quote) 界面细节是否到位，这部分通常需要强依赖后端的 API。
*   **多源归集与热钱包交互视图**：平台热钱包的充值状态监控，用户对自己在链上充值的 TxID 确认状态前端展现。

### C. 实体卡特有流程 (Physical Card Flow)
*   文档提到支持“实体卡”与“虚拟卡”。目前 `app/card/` 目录没有明确区分实体卡的**邮寄地址填写、物流追踪、实体卡激活 (Activation Code)** 的特定页面流程（目前只有统称的 `apply.tsx`）。

### D. 国际化与高级架构要求 (I18n & Architecture)
*   **国际化 (I18n)**：`package.json` 中未发现明确的国际化依赖库 (如 `i18next` 或 `react-intl`)。作为一个有出海/全球支付属性的Ｕ卡应用，多语言支持通常是必须的。
*   **错误监控与上报**：架构文档中提到了前端请求追踪 (RequestID)、以及安全防控。目前项目中未看到如 Sentry 或日志上报 SDK 的集成。

## 3. 建议的下一步开发重点

1.  **完善 API 层**：因为前端强依赖后端（Go/Node.js），需要检查 API Client 层（如 Axios/Fetch 结合 React Query）是否已经配置好相关的 Token 拦截、错误处理和 RequestID 传递。
2.  **补齐 KYC 模块**：这是新用户使用的第一个核心卡点。
3.  **开发实体卡邮寄与激活页面**：区分虚拟卡（秒开）和实体卡（要物流）的不同业务流。
4.  **补充多语言 (i18n)**：如果产品定位全球，应该尽早引入 i18n 框架免收后期重构之苦。
