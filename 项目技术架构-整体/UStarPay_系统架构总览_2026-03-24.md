# UStarPay 系统架构总览
更新时间：2026-03-24

## 1. 文档目标

这份文档回答 5 个问题：

1. `D:\snow` 下面每个仓库分别负责什么。
2. UStarPay 当前真实系统边界是什么。
3. 后端、数据库、后台前端、App、官网分别怎么分工。
4. 本地调试与测试服发布的运行模式有什么区别。
5. 当前代码层面还有哪些需要持续关注的架构风险点。

## 2. 工作区总览

当前主工作区：

```text
D:\snow
```

核心仓库分工如下：

| 路径 | 角色 | 当前定位 |
| --- | --- | --- |
| `D:\snow\ustarpay-backend-go` | Go 后端主仓 | UStarPay 核心业务后端 |
| `D:\snow\ustarpay-db` | 数据库脚本仓 | 基线 SQL、模块 SQL、增量 SQL、初始化脚本 |
| `D:\snow\ustarpay-frontend-admin` | 后台前端仓 | 平台后台、租户后台、共享前端能力 |
| `D:\snow\ustarpay-app` | App 仓 | 移动端和 App Web |
| `D:\snow\ustarpay-website` | 官网仓 | 官方展示站点 |
| `D:\snow\ustarpay-ops` | 运维编排仓 | 测试服部署、Dockerfile、编排和运维脚本 |
| `D:\snow\tgo` | 外部协同系统工作区 | 客服/IM 相关系统，不属于 UStarPay 主业务仓 |
| `D:\snow\docs` | 文档区 | 规则、SOP、需求梳理、技术文档 |

结论：

- UStarPay 不是单仓项目，而是多仓协作结构。
- 真正的主链路仓库是：后端、数据库、后台前端、App、Ops。
- TGO 与 UStarPay 有集成，但不是同一个系统代码仓。

## 3. 系统边界

### 3.1 UStarPay 主业务层

当前主业务主要覆盖：

- 用户体系
- KYC
- 钱包
- 充值 / 提现
- 卡业务
- 邀请返佣
- 支付充值
- 租户后台 / 平台后台

### 3.2 外部协同层

当前外部依赖主要包括：

- TGO 客服 / IM
- 发卡服务商
- Binance Pay
- 链上节点和资产网络

这意味着：

- UStarPay 本身不是“只做钱包”的系统
- 当前已经演进为“钱包 + 卡 + 支付 + 返佣 + 后台运营”的综合业务系统

## 4. 运行拓扑

当前主链路可以抽象为：

```text
App / 租户后台 / 平台后台 / 官网
        |
        v
      Kong
        |
        +--> auth-service
        +--> tenant-api
        +--> admin-service
        +--> tenant-admin-service
        +--> webhook-service
                  |
                  +--> PostgreSQL
                  +--> Redis
                  +--> NATS JetStream
                  +--> Jaeger
                  +--> 第三方支付 / 发卡商 / 外部回调
```

说明：

- `Kong` 是统一入口层
- 前端正常情况下不直接打 `1800x` 端口
- `webhook-service` 负责异步回调、事件处理、外部通知等链路

## 5. 后端架构

后端主仓：

```text
D:\snow\ustarpay-backend-go
```

### 5.1 分层结构

| 目录 | 作用 |
| --- | --- |
| `cmd` | 服务和工具入口 |
| `configs` | 按服务和环境拆分的配置 |
| `internal\adapters` | HTTP、DI、handler、route 等适配层 |
| `internal\domains` | 核心业务领域层 |
| `internal\infrastructure` | 仓储和数据库落地实现 |
| `pkg` | 公共基础组件 |

这说明当前后端不是把所有逻辑堆在 controller 里，而是偏向分层、领域化组织。

### 5.2 服务入口

| 服务 | 入口 | 说明 |
| --- | --- | --- |
| `auth-service` | `cmd\services\auth\main.go` | 登录、鉴权、权限基础能力 |
| `tenant-api` | `cmd\apis\tenant\main.go` | App 与租户业务 API |
| `admin-service` | `cmd\services\admin\main.go` | 平台后台 API |
| `tenant-admin-service` | `cmd\services\tenant-admin\main.go` | 租户后台 API |
| `webhook-service` | `cmd\workers\webhook\main.go` | 回调、异步通知、事件处理 |

另外还有几个工具入口：

- `cmd\migrate`
- `cmd\check-data`
- `cmd\list-db`

这些偏本地辅助，不是常驻服务。

### 5.3 主要领域模块

当前 `internal\domains` 中比较关键的业务域包括：

| 模块 | 职责 |
| --- | --- |
| `auth` | 登录、Token、鉴权、登录日志 |
| `tenant-user` | 用户、KYC、邀请关系、地址簿、用户账户 |
| `wallet` | 钱包地址、余额、链路钱包能力 |
| `deposit` | 充值记录和入账 |
| `withdrawal` | 提现申请、审核、处理和风控 |
| `fundflow` | 统一资金流水口径 |
| `card` | 开卡、卡记录、卡账户、卡消费、实体卡、卡商配置 |
| `invite-rebate` | 返佣配置、返佣档案、返佣事件、结算记录 |
| `payment` | Binance Pay 等支付充值能力 |
| `collection` | 自动归集配置、任务、子任务 |
| `tenant` | 租户级配置和面向租户的聚合业务 |
| `admin` | 平台侧管理能力 |

结论：

- 当前后端已经明显超出“单一钱包 API”的范围。
- 卡、返佣、支付、KYC、后台配置已经都是一等模块。

## 6. 数据库与 SQL 组织

数据库脚本仓：

```text
D:\snow\ustarpay-db
```

### 6.1 组织方式

| 目录或文件 | 用途 |
| --- | --- |
| `base` | 基础 schema 和兼容函数 |
| `admin` | 平台后台和 RBAC 相关 |
| `tenant` | 租户、用户、KYC、通知等 |
| `wallet` | 钱包和 KMS 相关 |
| `card` | 卡业务结构和专题文档 |
| `config` | 基础配置数据 |
| `init-local.sql` | 本地恢复风格脚本 |
| `init-schema-uuidv7.sql` | 当前结构基线参考 |
| `2026-03-xx_*.sql` | 日常增量变更脚本 |

### 6.2 当前 SQL 协作口径

必须坚持这几个原则：

1. 结构变更优先产出独立 SQL 脚本。
2. 测试服发布时，代码和 SQL 要同步。
3. SQL 不只是进仓库，还要实际执行到目标库。
4. 不要把某次历史 dump 当作长期主线。

## 9. 当前关键业务链路

### 9.1 钱包主链路

钱包相关当前主要围绕：

- 资产列表
- 资金账户
- 充值记录
- 提现申请
- 资金流水
- 地址簿
- 转卡

这套链路的对外展示口径，最终应尽量收敛到真实接口和真实资金流水。

### 9.2 卡业务主链路

卡业务当前主要覆盖：

- 开卡预检
- 普通开卡
- 实体卡
- 卡消费
- 卡转账 / 钱包转卡
- 卡敏感信息和安全能力

### 9.3 邀请返佣

当前返佣一期已明确走现有架构复用路线：

- 复用邀请关系
- 复用 KYC
- 复用购卡链路
- 复用资金账户和资金流水
- 复用配置和调度能力

当前后台规划为：

- `邀请返佣`
- `返佣配置`
- `专属档案`
- `返佣事件`
- `结算记录`

### 9.4 Binance Pay 充值

该需求本质是商户代收充值链路，主流程是：

1. App 创建充值单
2. 后端调用 Binance Pay 下单
3. 用户拉起 Binance Pay 支付
4. Webhook 回调确认
5. 本地订单入账
6. App 轮询或刷新订单状态

这条链路当前不是从零开始，而是在现有半成品上继续补用户侧闭环。

## 10. 当前架构关注点

### 10.1 已经纠偏的重点

卡和返佣相关主链路，近期已经按“有资金出入的关联业务要收在同一 DB 事务内”的要求做过事务收口，至少覆盖了：

- 普通开卡成功与返佣
- 预制实体卡审核通过与返佣
- 卡消费结算与返佣

### 10.2 仍需持续关注的点

1. 卡 `precheck` 与真实 `apply` 的 KYC 口径仍不完全一致
2. `collection` 模块还有“先冻结、后建任务、失败再补偿”的链式模式
3. 钱包里仍有个别双表顺序冻结点，需要继续确认是否为主入口
4. 测试服验证很依赖真实配置，不要把环境阻塞误判成代码阻塞
5. App Web 和前端构建容易受环境变量影响，配置漂移要重点盯

## 11. 与 TGO 的关系

`D:\snow\tgo` 不是 UStarPay 的一个子模块，而是独立工作区。

当前更合理的理解是：

- UStarPay：钱包、卡、KYC、返佣、支付、后台
- TGO：客服、IM、会话协同等外部系统

两者通过配置和接口集成，而不是一个单仓前后端项目。

## 13. 一句话结论

UStarPay 当前已经是一个多仓、多服务、强配置驱动的业务系统。后续所有开发和排障，都不能只盯某一个仓库或某一个页面，而要同时看：

- 代码分层
- 配置覆盖
- SQL 变更
- 网关路由
- 测试服真实环境
