# UStarPay 本地启动手册
更新时间：2026-03-24

## 1. 适用范围

本手册只解决一件事：在 `D:\snow` 工作区内，如何把 UStarPay 的本地联调环境稳定拉起来。

适用仓库：

- `D:\snow\ustarpay-backend-go`
- `D:\snow\ustarpay-db`
- `D:\snow\ustarpay-frontend-admin`
- `D:\snow\ustarpay-app`
- `D:\snow\ustarpay-website`
- `D:\snow\ustarpay-ops`

不展开 TGO 内部源码细节，TGO 只按外部协同系统对待。

## 2. 推荐启动口径

当前最稳的本地联调方式不是“一把梭全部容器化”，而是：

1. 基础设施和 Kong 网关用 Docker 启动。
2. Go 业务服务按服务入口分别本机启动。
3. 前端、App、官网各自按 Vite 或 Expo 启动。

这样做的原因：

- 更接近当前团队真实开发方式。
- 后端单服务调试更方便。
- 避免把历史残留 compose 误当成唯一权威入口。

## 3. 启动前准备

建议本机至少具备以下环境：

- Go：以仓库 `go.mod` 为准，当前声明 `go 1.24`，`toolchain go1.25.8`
- Node.js：建议 `>= 18`
- npm：建议 `>= 8`
- Docker Desktop

当前相关仓库脚本要求：

- `tenant-portal` 要求 `node >= 16`
- `admin-platform` 使用 Vite + React + TypeScript
- `ustarpay-app` 使用 Expo Router

## 4. 工作区与仓库职责

| 路径 | 作用 |
| --- | --- |
| `D:\snow\ustarpay-backend-go` | Go 后端主仓，承载认证、App API、后台 API、Webhook |

## 5. 本地基础设施启动

当前推荐入口：

```text
D:\snow\docker-compose.test.yml
```

它至少能提供本地联调所需的核心基础设施：

- PostgreSQL
- Redis
- NATS
- Jaeger
- Kong

推荐启动命令：

```powershell
cd D:\snow
docker compose -f .\docker-compose.test.yml up -d kong-database go-services-database go-services-redis nats jaeger kong-bootstrap kong
```

如果你想直接把本地后端容器也一起起起来，也可以执行：

```powershell
cd D:\snow
docker compose -f .\docker-compose.test.yml up -d
```

但日常开发仍更推荐“基础设施用 compose，业务服务用 `go run`”。

### 5.1 本地默认端口

| 服务 | 端口 |
| --- | --- |
| Kong Proxy | `8000` |
| Kong Admin API | `8001` |
| Kong Manager | `8002` |
| PostgreSQL | `5432` |
| Redis | `6379` |
| NATS | `4222` |
| NATS HTTP | `8222` |

## 6. 本地数据库初始化

数据库脚本仓：

```text
D:\snow\ustarpay-db
```

当前要点：

1. `go-services-database` 容器默认只会起一个空的 `nh_wallet` 库，不会自动灌完整业务表。
2. 第一次本地启动时，需要手动执行 SQL。
3. 日常协作不能只靠旧 dump，要同步执行日期增量脚本。

### 6.1 常见 SQL 入口

| 文件 | 用途 |
| --- | --- |
| `init-local.sql` | 更像本地完整恢复脚本 |
| `init-schema-uuidv7.sql` | 更适合作为当前结构基线参考 |
| `2026-03-xx_*.sql` | 当天或近期增量变更，发布和本地联调都要关注 |

### 6.2 执行建议

如果你要快速恢复一套完整本地库，优先看：

- `D:\snow\ustarpay-db\init-local.sql`

如果你要看当前结构基线，再补业务增量，优先看：

- `D:\snow\ustarpay-db\init-schema-uuidv7.sql`
- `D:\snow\ustarpay-db\2026-03-xx_*.sql`

原则：

- 结构变更优先沉淀独立 SQL。
- 本地联调和测试服发布都要同步 SQL，不只是把脚本提交进仓库。

## 7. Go 后端启动

后端仓库：

```text
D:\snow\ustarpay-backend-go
```

当前真实服务入口：

| 服务 | 入口 | 默认端口 |
| --- | --- | --- |
| `auth-service` | `cmd\services\auth` | `18001` |
| `tenant-api` | `cmd\apis\tenant` | `18002` |
| `admin-service` | `cmd\services\admin` | `18003` |
| `tenant-admin-service` | `cmd\services\tenant-admin` | `18004` |
| `webhook-service` | `cmd\workers\webhook` | `18005` |

### 7.1 推荐启动方式

先安装依赖：

```powershell
cd D:\snow\ustarpay-backend-go
go mod tidy
```

然后分别开终端启动服务。PowerShell 示例：

```powershell
cd D:\snow\ustarpay-backend-go
$env:APP_ENV = "development"
go run .\cmd\services\auth
```

```powershell
cd D:\snow\ustarpay-backend-go
$env:APP_ENV = "development"
go run .\cmd\apis\tenant
```

```powershell
cd D:\snow\ustarpay-backend-go
$env:APP_ENV = "development"
go run .\cmd\services\admin
```

```powershell
cd D:\snow\ustarpay-backend-go
$env:APP_ENV = "development"
go run .\cmd\services\tenant-admin
```

```powershell
cd D:\snow\ustarpay-backend-go
$env:APP_ENV = "development"
go run .\cmd\workers\webhook
```

### 7.2 最小启动组合

| 场景 | 至少需要的服务 |
| --- | --- |
| App 登录、钱包、卡、充值提现 | `auth-service` + `tenant-api` |
| 租户后台联调 | `auth-service` + `tenant-admin-service` |
| 平台后台联调 | `auth-service` + `admin-service` |
| 异步回调、结算、支付回调排查 | 额外加 `webhook-service` |

### 7.3 配置加载口径

配置目录：

```text
D:\snow\ustarpay-backend-go\configs
```

当前规则：

1. 公共配置在 `configs\common\{env}.yml`
2. 服务私有配置在 `configs\<service>\{env}.yml`
3. 环境变量可以覆盖配置文件

排查问题时，不要只看代码，还要同时看：

- `main.go`
- 对应 `configs`
- 当前终端环境变量

- App Web 构建连到哪个环境，取决于构建时注入的环境变量
- 不要只看代码默认值，实际以构建时注入为准

## 11. 联调顺序建议

推荐按下面顺序起：

1. 起 Docker 基础设施和 Kong
2. 准备本地数据库
3. 起 `auth-service`
4. 根据场景再起 `tenant-api` / `tenant-admin-service` / `admin-service` / `webhook-service`
5. 起后台前端、App 或官网

## 12. 常见问题

### 12.1 前端能打开，但接口全 404 或 502

优先检查：

1. `http://localhost:8000` 的 Kong 是否起来
2. 对应 Go 服务是否真的在本机启动
3. 前端代理目标是否还是 `localhost:8000`

### 12.2 App Web 打到错误环境

优先检查：

1. 构建或启动前是否设置了 `EXPO_PUBLIC_API_BASE_URL`
2. 是否重新执行了 `npm run web` 或 `npm run build`
3. 是否误把旧构建产物当成新包

### 12.3 本地库表不全

优先检查：

1. 是否只起了 PostgreSQL，但没执行 SQL
2. 是否漏执行当天增量脚本
3. 是否把历史 `dump` 当成最新结构

### 12.4 `Makefile` 能不能直接当唯一入口

不建议。

当前仓库里的 `Makefile` 明显带有早期模板和历史残留，不适合直接当作现阶段唯一启动规范。现阶段仍以：

- `go run` 真实服务入口
- `configs` 真实配置
- 当前 compose 编排文件

这三者交叉确认为准。

## 13. 测试服入口补充

测试服不在本手册内展开细节，只记录最关键入口：

- 部署根目录：`/opt/snow`
- 编排文件：`D:\snow\ustarpay-ops\deploy\staging\docker-compose.ustarpay.yml`

测试服发布时，默认动作必须包含：

1. 同步代码
2. 同步并执行 SQL
3. 重建对应服务
4. 做基本冒烟验证

## 14. 一句话结论

本地联调最稳的方式就是：

- 基础设施走 `docker-compose.test.yml`
- 业务服务走 `go run`
- 前端和 App 各自按项目启动
- SQL 手动确认并同步到当前环境
